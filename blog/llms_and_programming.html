LLMs and programming

I wrote this blog post after reading this tweet by François Chollet. Originally, I wanted to write a few tweets but realized afterwards that I actually want to write a lot more than would fit into Twitter’s maximum tweet lengths. The context for the tweet and this blog post is that a startup recently gave a demo of an “AI software engineer” which could perform a few nontrivial programming tasks. As of the time of writing, the tweet thread read:

If you could fully automate software engineering (my job), I think that would be great, since I could then move on to higher-leverage things. Making software is a means to an end, not the end (software is different from art in that sense).

However I have not yet seen any signs that we’re going to be able to automate more than an infinitesimal fraction of my job – not even “this might work someday maybe if $100B gets invested in the area and if we’re very lucky” type of signs. I wish!

The thing is, software engineering is not about copy/pasting code. Software engineering is the development and manipulation of mental models of problems and their solutions. It’s thinking and problem solving.

If you had a machine that could craft mental models, you’d have AGI.

There’s a lot of confusion in this area, because many people are under the impression that software is in fact about copy/pasting random code snippets – which of course sounds highly automatable if you have an interpolative database of all software ever written.

And maybe for some devs that’s what it is! But the way I see it, that kind of development is to software engineering what boilerplate copyediting is to storytelling.

Code is largely worthless, more of a liability than an asset. Problem-solving is where the value is. (source).

Now, I’m not entirely certain if this sentiment will withstand the test of time. While I believe that for humans, programming is equivalent to a relatively high degree of intelligence, I also think that AIs don’t necessarily need to reach that level, or at least, their current trajectory of intelligence (which likely falls short of “AGI,” whatever that means) will not require such high intelligence. This is because LLM intelligence is quite peculiar and superhuman in certain areas. With long context lengths (i.e., 200k+), AIs can juggle far more things than humans can. As humans, we are fairly memory-limited; we can only think of a limited number of things at one time and have to rely on less reliable long-term memory abstractions for the rest. An LLM with a sufficiently long context does not have to deal with any of that. An example of this is translation: a recent demo of Gemini showcased it translating the Kalamang language (which it wasn’t trained on) just as well as a human who spent twenty hours with the same 573-page book and dictionary. It accomplished this in mere minutes. This sort of behavior is intelligent but in a way that is completely alien to how humans learn and carry out translation.

A more concrete hypothetical: if we reach the point where we can pass the entire Linux kernel source code into an LLM’s context, how many mental models or abstractions does it really need to use to fix a bug or write code that implements a feature? I am genuinely uncertain about that. It is very alien for us humans to imagine being able to juggle this magnitude of complexity right in our heads or what it might enable AI to do, even without “AGI.” In any case, the demonstrations we have today strongly indicate that this is a capability AI has over humans. Of course, all of this rests on the assumption that we’ll be able to scale the context length of these models to, at some point, hold 27 million lines of code. This is a formidable problem, but I have not seen any indicators that scaling will stop. LLMs have already grown by more than three orders of magnitude in the span of a few years, and they might continue to grow like that.

Nobody can predict the future, and I think nobody my age or older envisioned this situation when starting a degree in software engineering. It is normal for many engineers to experience anxiety over this: software engineering takes a long time to learn and is such a hyperspecialized endeavor. If it is automated, switching to other jobs will be challenging. Of course, various other jobs have been automated away, and the people who worked these jobs undoubtedly felt anxiety while this process occurred—automation has mostly worked out for the better for humanity, as we would never be able to support six billion humans on this planet without it. I enjoy programming, and one of the main reasons I appreciate my research area is that I get to do two things I love: programming and mathematics. I strongly believe automation is, in the aggregate, good for us, but on a personal level, I still can’t help but feel anxious about this. If software engineer isn’t automated, as the tweet suggests, I think I might be a lot less anxious about the future.
